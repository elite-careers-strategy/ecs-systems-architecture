# Architecture Overview

This document describes the public-safe version of the Elite Careers Strategy operating architecture.

## Purpose

ECS is designed as a system-led advisory firm. The system exists to make the candidate journey clearer, more measurable and more governed.

The operating model is built around five principles:

1. Diagnose before prescribing.
2. Keep proof consent-safe.
3. Turn repeated work into reusable systems.
4. Preserve founder judgement for decisive moments.
5. Keep public claims evidence-backed and privacy-safe.

## System layers

### 1. Public surface

The public surface explains what ECS is, who it serves and how a candidate or family enters the diagnostic path.

Public surfaces can include:

- Website pages
- Articles
- Founder profile
- Public framework explainers
- Public architecture notes

Public-surface rule: no claim should appear unless it is safe, accurate and deployable.

### 2. Intake and qualification

The intake layer captures the context needed to understand whether ECS can help.

Typical inputs include:

- Target sector
- Target geography
- Candidate stage
- Timeline
- Current bottleneck
- Decision-maker involvement

The purpose is not volume. The purpose is fit.

### 3. Diagnostic and route design

ECS diagnoses where the candidate is blocked and designs the support structure around that gap.

The public diagnostic logic is simple:

- Stage 1 - getting in
- Stage 2 - getting through
- Stage 3 - getting offers

Each stage has different failure modes and therefore different fixes.

### 4. Client operating system

Once engaged, the candidate journey should be visible and structured.

The client operating system tracks:

- Target firms
- Application pipeline
- Written assets
- Interview readiness
- Assessment-centre readiness
- Feedback loops
- Next actions

The goal is to make progress legible.

### 5. Proof and privacy governance

Proof is valuable only if it is accurate and deployable.

The governance layer controls:

- What can be said
- Who can be named
- Which evidence is private
- Which proof is public-safe
- Which claims require further verification

## Human and system boundary

Systems support repeatability. Humans retain judgement.

System-supported work:

- Drafting
- Reminders
- Routing
- Checklists
- Readiness tracking
- Claim checking

Human-owned work:

- Sensitive judgement
- Client-facing trust moments
- Pricing discussion
- Public positioning
- Final approval of proof deployment

## Public boundary

This public architecture does not expose internal implementation details, credentials, database IDs, client evidence or private operating prompts.
